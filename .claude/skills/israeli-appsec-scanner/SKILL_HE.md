# סורק אבטחת אפליקציות ישראלי

כלי סריקה ובדיקות אבטחה שמותאם לאפליקציות ווב ישראליות. הסקיל מכסה את כל הספקטרום, מ-OWASP Top 10 ועד עמידה בדרישות רשות הגנת הפרטיות, עם דגש על וקטורי תקיפה שקשורים לעברית ו-RTL.

> **הערת גרסה ל-OWASP Top 10 (2025-2026).** הרשימה למטה מבוססת על OWASP Top 10 משנת 2021. בשנת 2025 פורסם OWASP Top 10 חדש שמשנה את המספור: A01 כולל עכשיו גם SSRF (אין יותר A10 נפרד ל-SSRF), A03 הוא **Software Supply Chain Failures** חדש, A05 הוא Injection (ירד מ-A03), A09 שונה ל-**Security Logging and Alerting Failures**, ויש A10 חדש על **Mishandling of Exceptional Conditions**. בעת ביקורת ב-2026, השתמשו במספור 2021 כפיגום עבודה והעבירו את הממצאים לרשימה של 2025 בכתובת https://owasp.org/Top10/2025/ לפני הדוח.

## רשימת בדיקות OWASP Top 10 (בהקשר ישראלי)

עברו על כל קטגוריה בשיטתיות. לכל ממצא, סמנו רמת חומרה (קריטי/גבוה/בינוני/נמוך) ותנו המלצת תיקון.

### A01: בעיות בבקרת גישה

- [ ] תוודאו שכל נקודות ה-API דורשות אימות (בדקו middleware של Next.js, guards של NestJS)
- [ ] תוודאו שבקרת גישה מבוססת תפקידים מכסה admin, user ואנונימי
- [ ] בדקו שנתיבי URL בעברית לא יכולים לעקוף הרשאות מבוססות ניתוב
- [ ] חפשו IDOR על משאבים שגלויים למשתמשים
- [ ] תוודאו שהגדרות CORS מגבילות את ה-origins לדומיינים הישראליים הצפויים
- [ ] תוודאו שטוקני JWT נבדקים בצד השרת, לא רק בצד הלקוח
- [ ] בדקו directory traversal עם קטעי נתיב מקודדים בעברית (%D7%90 וכו')

### A02: כשלים קריפטוגרפיים

- [ ] תוודאו TLS 1.2+ על כל החיבורים (ספקי אחסון ישראליים לפעמים משתמשים בגרסאות ישנות כברירת מחדל)
- [ ] בדקו שסיסמאות מוצפנות עם bcrypt/argon2 (לא MD5/SHA1)
- [ ] תוודאו שמספרי תעודת זהות מוצפנים ב-rest
- [ ] תוודאו שנתוני כרטיסי אשראי לא נשמרים מקומית (דרישת PCI DSS)
- [ ] בדקו שמפתחות service role של Supabase לא חשופים ללקוח
- [ ] תוודאו שטוקנים של שערי תשלום ישראליים (קארדקום, טרנזילה) מאוחסנים בצורה מאובטחת

### A03: הזרקות (Injection)

- [ ] בדקו SQL injection עם טקסט בעברית (`'; DROP TABLE --` עם תווים בעברית)
- [ ] תוודאו שימוש ב-parameterized queries לכל פעולות מסד הנתונים עם טקסט עברי
- [ ] בדקו NoSQL injection בשאילתות MongoDB (נפוץ בסטארטאפים ישראליים שמשתמשים ב-MEAN stack)
- [ ] בדקו command injection דרך שמות קבצים שמכילים תווים בעברית
- [ ] תוודאו הגנה מפני LDAP injection אם יש אינטגרציה עם ספריות ארגוניות ישראליות
- [ ] בדקו template injection בתבניות אימייל עם תוכן בעברית

### A04: תכנון לא מאובטח

- [ ] תוודאו rate limiting על נקודות קצה של שערי SMS ישראליים (למנוע הצפה ב-OTP)
- [ ] חפשו פגמים בלוגיקה העסקית בתהליכי תשלום ישראליים
- [ ] תוודאו שטפסים מרובי שלבים (נפוצים באינטגרציות עם שירותי ממשלה) שומרים state בצורה מאובטחת
- [ ] תוודאו שמגבלות אורך קלט בעברית מתחשבות בתווי UTF-8 מרובי בתים

### A05: תצורה לא מאובטחת

- [ ] בדקו כותרות HTTP של אבטחה (CSP, X-Frame-Options, HSTS)
- [ ] תוודאו שדפי שגיאה לא חושפים stack traces או הודעות דיבאג בעברית
- [ ] תוודאו שסיסמאות ברירת מחדל הוחלפו בכל השירותים
- [ ] בדקו שקבוצות אבטחה של ספקי ענן ישראליים (AWS IL region, GCP) מוגדרות כראוי
- [ ] תוודאו שמדיניות RLS של Supabase מופעלת על כל הטבלאות
- [ ] תוודאו שקבצי `.env` עם credentials של שירותים ישראליים נמצאים ב-`.gitignore`

### A06: רכיבים פגיעים ומיושנים

- [ ] הריצו `npm audit` או `pnpm audit` וטפלו בממצאים קריטיים/גבוהים
- [ ] חפשו פגיעויות ידועות בספריות ספציפיות לעברית (hebrew-date, מעבדי טקסט דו-כיווני)
- [ ] תוודאו שתמונות בסיס של קונטיינרים מעודכנות (עם `trivy image`)
- [ ] בדקו גרסאות מיושנות של SDK של שערי תשלום ישראליים

### A07: כשלי זיהוי ואימות

- [ ] תוודאו ולידציה של פורמט מספר טלפון ישראלי (+972) בתהליכי אימות
- [ ] בדקו שקודי OTP למספרים ישראליים פגים אחרי 5 דקות לכל היותר
- [ ] בדקו שמדיניות סיסמאות תומכת בתווים בעברית
- [ ] תוודאו ניהול סשנים תקין אחרי אימות עם ספקי OAuth ישראליים
- [ ] בדקו הגנה מפני credential stuffing על נקודות קצה של התחברות

### A08: כשלי שלמות תוכנה ונתונים

- [ ] תוודאו שלמות חבילות npm עם ולידציית lockfile
- [ ] בדקו אבטחת pipeline של CI/CD (GitHub Actions, Vercel deployment hooks)
- [ ] תוודאו שוובהוקים של שירותים ישראליים צד שלישי מאומתים (ולידציית חתימה)
- [ ] תוודאו Subresource Integrity (SRI) לסקריפטים חיצוניים

### A09: כשלי לוגים וניטור

- [ ] תוודאו שאירועי אימות נרשמים (התחברות, התנתקות, ניסיונות כושלים)
- [ ] בדקו שלוגים קולטים טקסט בעברית נכון (קידוד UTF-8)
- [ ] תוודאו שנתיבי ביקורת שנדרשים על ידי רשות הגנת הפרטיות מתוחזקים לגישה למידע אישי
- [ ] תוודאו שאחסון לוגים עומד בדרישות שמירת מידע של החוק הישראלי

### A10: זיוף בקשות בצד השרת (SSRF)

- [ ] תוודאו ולידציית URL לכתובות שהמשתמש מספק (חסימת טווחי רשת פנימיים)
- [ ] בדקו שאינטגרציות עם ממשקי API ממשלתיים ישראליים מוודאות URL-ים של הפניות
- [ ] בדקו SSRF דרך URL-ים מקודדים בעברית ותקיפות IDN homograph

## עמידה בדרישות רשות הגנת הפרטיות

חוק הגנת הפרטיות (1981) והתקנות שלו מטילים דרישות ספציפיות על אפליקציות שמטפלות במידע אישי של תושבי ישראל. תיקון 13 (בתוקף מ-14 באוגוסט 2025) דורש דיווח מיידי לרשות על אירוע אבטחה חמור, ממונה הגנת מידע (DPO) במקרים מסוימים, הרחבת ההגדרות של "מידע אישי" ו"מידע רגיש", חובת הודעה למאגרים עם מידע רגיש במיוחד על מעל 100,000 אנשים, ופיצוי סטטוטורי עד 100,000 ש"ח ללא הוכחת נזק (רובד נמוך יותר של 10,000 ש"ח חל על הפרות הודעה / זכויות נושא מידע).

### רישום מאגרי מידע (שונה בתיקון 13)

תיקון 13 (אוגוסט 2025) ביטל את חובת הרישום הרחבה. הכלל הישן, שחייב רישום של כל מאגר עם 10,000+ רשומות, או מידע רגיש, או שימוש לדיוור ישיר, כבר לא חל. הכללים הנוכחיים:

1. **רישום** נדרש כיום רק עבור: (א) **סוחרי מידע**, כלומר מאגרים שמטרתם העיקרית היא איסוף מידע אישי כדי להעבירו לאחרים כעסק, כשהמאגר מכיל מידע על יותר מ-10,000 אנשים; ו(ב) **גופים ציבוריים**. סף ה-10,000 שורד רק בהקשר של סוחרי מידע, לא כסף כללי.
2. **הודעה (לא רישום)**: בעל מאגר שמכיל "מידע רגיש במיוחד" על יותר מ-100,000 אנשים חייב להודיע לרשות על פרטי זהות, קשר ו-DPO, גם כשאין חובת רישום.
3. **אין חידוש שנתי**: חובת החידוש השנתי בוטלה. מאגר שנרשם בעבר ושכבר לא עומד בתנאים אינו נמחק אוטומטית, ובעל המאגר צריך לבקש מחיקה מהפנקס.
4. רוב אפליקציות הצרכן הרגילות כבר לא צריכות להירשם בכלל, אבל עדיין חייבות בכל חובות האבטחה, ההסכמה וזכויות נושא המידע שלהלן.

```
    - השיבו תוך 30 יום לבקשות עיון במידע
    - ספקו מידע בפורמט מובנה וקריא מכונה
    - תמיכה בעברית בכל התקשורת מול נושאי מידע

    - "אירוע אבטחה חמור" חייב דיווח לרשות
    - ההודעה צריכה להיות בעברית לתושבי ישראל
    - תעדו את כל האירועים ושלבי התיקון
```

### העברת מידע חוצת גבולות

החוק הישראלי מגביל העברת מידע אישי מחוץ לישראל. ההעברה מותרת כש:

- מדינת היעד מספקת רמת הגנה נאותה (מדינות האיחוד האירופי, שמוכרות על ידי הרשות)
- נושא המידע הסכים להעברה
- ההעברה דרושה לקיום חוזה
- יש ערובות חוזיות מתאימות (סעיפים חוזיים סטנדרטיים)

**תרחישים נפוצים באפליקציות ישראליות:**
- אחסון בענן ב-AWS/GCP: תוודאו שיש הסכם עיבוד מידע
- שימוש בכלי SaaS אמריקאיים: תוודאו ערובות חוזיות
- שירותי אנליטיקה: שקלו אנונימיזציה של מידע לפני ההעברה

## סריקת תלויות

### npm/pnpm audit

```bash
# סריקת פגיעויות
pnpm audit

# תיקון אוטומטי היכן שאפשר
pnpm audit --fix

# יצירת דוח JSON מפורט
pnpm audit --json > audit-report.json

# בדיקה לפי רמת חומרה
pnpm audit --audit-level=high
```

### סריקת קונטיינרים עם Trivy

> **פריצת שרשרת אספקה של Trivy (מרץ 2026).** התגיות של `aquasecurity/trivy-action` ו-`setup-trivy` נפרצו: גרסת v0.69.4 זדונית (19 במרץ 2026) ואחריה תמונות Docker זדוניות v0.69.5 ו-v0.69.6 (GHSA-69fq-xp46-6x23 / CVE-2026-33634). הצמידו לגרסה נקייה **v0.70.x ומעלה** (הימנעו מכל הטווח v0.69.4 עד v0.69.6), הצמידו actions ל-SHA מלא של commit, ואמתו checksums.

```bash
# סריקת תמונת Docker
trivy image your-app:latest

# סריקה עם סינון חומרה
trivy image --severity HIGH,CRITICAL your-app:latest

# סריקת מערכת קבצים (לפרויקטים מקומיים)
trivy fs --security-checks vuln,secret,config .
```

```bash
# עם pip-audit
```

## זיהוי סודות

### סריקת credentials של שירותים ישראליים

אפליקציות ישראליות משתמשות בדרך כלל ב-credentials של שירותים שלעולם לא צריכים להיכנס ל-version control:

**שערי תשלום ישראליים:**
- קארדקום: מספרי מסוף, שמות משתמש API, קודי פעולה
- טרנזילה: קודי ספק, סיסמאות מסוף
- PayMe: מזהי מוכר, מפתחות API
- משולם: קודי עמוד, מפתחות API

**שירותים ישראליים:**
- מפתחות API של דואר ישראל
- credentials של ממשקי API בנקאיים (פועלים, לאומי, דיסקונט, מזרחי)
- טוקנים של Gov.il API
- מפתחות API של שערי SMS ישראליים (019, Cellact, InforUMobile)

### שימוש ב-TruffleHog

```bash
# סריקת היסטוריית git
trufflehog git file://. --only-verified

# סריקת ענף ספציפי
trufflehog git file://. --branch main --only-verified

# סריקה עם פלט JSON
trufflehog git file://. --json > secrets-report.json
```

### שימוש ב-Gitleaks

```bash
# סריקת מאגר
gitleaks detect --source . --verbose

# סריקה עם קונפיגורציה מותאמת לשירותים ישראליים
gitleaks detect --source . --config .gitleaks.toml

# יצירת דוח
gitleaks detect --source . --report-format json --report-path gitleaks-report.json
```

## אבטחה ספציפית לעברית/RTL

### תקיפות טקסט דו-כיווני (Trojan Source)

תווי בקרה דו-כיווניים יכולים לגרום לקוד להיראות שונה ממה שהוא באמת עושה. זה רלוונטי במיוחד בבסיסי קוד מעורבים עברית/אנגלית.

**תווים מסוכנים שצריך לזהות:**
- U+202A (Left-to-Right Embedding)
- U+202B (Right-to-Left Embedding)
- U+202E (Right-to-Left Override)
- U+2066 עד U+2069 (Isolate characters)

**מניעה:** הריצו את סקריפט הזיהוי ב-`scripts/security-audit-checklist.py` כדי לסרוק קבצי קוד לתווים דו-כיווניים חבויים.

### תקיפות הומוגליף בעברית

תווים עבריים מסוימים נראים ויזואלית כמו תווים לטיניים, מה שפותח אפשרות לפישינג וזיוף:

| עברית | דומה ללטינית | Unicode |
|-------|-------------|---------|
| ס (סמך) | o | U+05E1 |
| ו (וו) | l, 1 | U+05D5 |
| ח (חת) | n | U+05D7 |
| ן (נון סופית) | l | U+05DF |

**מניעה:** נרמלו ובדקו תקינות של כל ה-URL-ים והמזהים שגלויים למשתמשים. דחו מחרוזות מעורבות כתב בהקשרים רגישים מבחינת אבטחה.

### ולידציית קלט בעברית

```javascript
// ולידציה לקלט בעברית בלבד
const HEBREW_PATTERN = /^[\u0590-\u05FF\s\-'".,:;!?()]+$/;

// ולידציה לקלט מעורב עברית/אנגלית
const MIXED_PATTERN = /^[\u0590-\u05FFa-zA-Z0-9\s\-'".,:;!?()@#$%&*]+$/;

// ולידציית מספר טלפון ישראלי
const IL_PHONE_PATTERN = /^(\+972|0)(5[0-9]|7[2-9])\d{7}$/;

// ולידציית מספר תעודת זהות עם ספרת ביקורת
function validateIsraeliId(id) {
  if (!/^\d{9}$/.test(id)) return false;
  let sum = 0;
  for (let i = 0; i < 9; i++) {
    let digit = parseInt(id[i]) * ((i % 2) + 1);
    if (digit > 9) digit -= 9;
    sum += digit;
  }
  return sum % 10 === 0;
}
```

## מלכודות נפוצות

- חוק הגנת הפרטיות עבר רפורמה בתיקון 13 (בתוקף מ-14 באוגוסט 2025). אירוע אבטחה חמור חייב להיות מדווח לרשות **מיד** (החוק אומר "מיד", לא דדליין קבוע של 72 שעות בנוסח GDPR), בנוסף להודעה לנפגעים כשיש סיכון גבוה. הפיצוי הסטטוטורי הוא כעת עד 100,000 ש"ח ללא הוכחת נזק. סוכנים שמצטטים ניסוח שלפני 2025 מפספסים את החשיפה החדשה.
- תווי UTF-8 בעברית תופסים 2 בתים כל אחד. ולידציית אורך קלט שסופרת בתים במקום תווים תדחה קלט עברי תקין באורך חצי מהצפוי.
- מספרי תעודת זהות ישראליים משתמשים בווריאנט של אלגוריתם Luhn לספרת ביקורת, לא Luhn סטנדרטי. סוכנים עלולים לממש ולידציה שגויה ולקבל מספרים לא תקינים.
- רישום מאגרי מידע צומצם דרמטית בתיקון 13 (אוגוסט 2025): הכלל הישן של "לרשום כל מאגר עם 10,000+ רשומות" בוטל. רישום חל כיום רק על סוחרי מידע (מאגרים מעל 10,000 אנשים שעיסוקם מכירת מידע) וגופים ציבוריים, עם חובת הודעה נפרדת למאגרים רגישים במיוחד מעל 100,000 אנשים. סוכנים שמצטטים את סף ה-10,000 הכללי הישן אינם מעודכנים.
- תקיפות טקסט דו-כיווני (BiDi) מסוכנות במיוחד בבסיסי קוד מעורבים עברית/אנגלית. סורקי אבטחה סטנדרטיים לא מזהים תווי RTL override (U+202E) שיכולים להסוות קוד זדוני.
